Step 05
Trigger OAuth — Connect Your Google Account
Pick any of the options below. All three do the same thing: open Google sign-in, save a token locally for future runs.
What is reauthenticate, and why do you need to call it?

reauthenticate is a tool built into the gsc-deep MCP server. When you call it, it does three things, in order:

  1. Opens a browser window to Google's OAuth consent page, using the OAuth client ID from your client_secrets.json
  2. After you click Allow, exchanges the authorization code Google sends back for an access token and a refresh token
  3. Saves both tokens to your local user-config directory so future tool calls don't need a re-login (the refresh token auto-renews the access token in the background)

When you must run it:

  • First-time setup — mandatory. No tokens exist yet, so any other gsc-deep tool you call will fail with "tool timed out" or "no credentials".
  • Switching the gsc-deep server to a different Google account. Saved tokens are single-account. Re-running reauthenticate and signing in with a different Gmail replaces the previous token.
  • If you suddenly get 401 / timeout errors after months of working fine. The refresh token can be revoked by Google (e.g. password changes, security review, or 6+ months of inactivity). Re-running fixes it.
After the first successful login you do not need to re-run reauthenticate on every Claude restart. Tokens persist on disk across sessions, and the refresh token keeps them valid indefinitely until revoked.
Option A — In Claude Desktop chat (recommended — one click, no terminal)
Claude Desktop — type in chat
Important: Sometimes typing just Call reauthenticate does not trigger the correct gsc-deep tool directly, especially if both MCPs are active. Always be explicit and tell Claude which server to use:
claude prompt — use this exact wording
Call reauthenticate gsc-deep
If the shorter version Call reauthenticate does not open a browser window within a few seconds, use the explicit version above. Specifying gsc-deep routes the call to the correct server every time.

  2. Click Allow on the permissions screen
  3. Wait for "Authentication successful" to appear in the browser before closing it
  4. gsc-deep is now connected. You do not need to include the account email in any future prompts — just mention the domain.

Complete the sign-in, wait for "Authentication successful", then come back to Command Prompt and press Ctrl + C to stop the server. Then restart Claude Desktop from the system tray so the saved token is loaded by the in-app server.

Option C — "Paste a URL in incognito" — not available for gsc-deep

gsc-multi offers this option because it uses a bundled OAuth client and a fixed callback URL (http://localhost:3000/oauth2callback) — both known ahead of time, so a working URL can be generated for you to paste anywhere.

gsc-deep uses your own OAuth client from client_secrets.json, and the underlying Python library binds the callback to a randomly-chosen localhost port per run. Neither value is known until the server is actually running, so a paste-URL can't be pre-generated reliably.

How to control which Google account gets picked anyway:
  • Easiest — sign out of all other Google accounts in your default browser before clicking Option A. Or open Claude Desktop in an environment where Chrome/Edge has only the target Gmail signed in.
  • Most reliable — in Step 02, add only the Gmail you want gsc-deep bound to as a test user. Google's account chooser will hide accounts that aren't whitelisted.
  • If you authenticated the wrong account — just call reauthenticate again with the right browser state. The new token overwrites the old one.

  3. Open a fresh PowerShell and re-run gcloud --version
Error 03 — PowerShell scripts disabled
File cannot be loaded because running scripts is disabled on this system
PSSecurityException — gcloud.ps1, npm, pipx etc. — UnauthorizedAccess
Fix
PowerShell execution policy is too restrictive — relax it for your user only
powershell — one-time fix
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned -Force

RemoteSigned is Microsoft's recommended developer default — local scripts run, only unsigned scripts downloaded from the internet are blocked.

Error 04 — Google "This app is blocked"
This app is blocked — Google blocked this access
During OAuth: "This app tried to access sensitive info in your Google Account. To keep your account safe, Google blocked this access."
Fix
Your Gmail isn't a test user on the OAuth consent screen, OR you picked the wrong account
  1. Go to console.cloud.google.com/apis/credentials/consent with the project that owns client_secrets.json selected
  2. Scroll to Test users → + Add Users → add your Gmail → Save
  3. Sign out of all other Google accounts at accounts.google.com/Logout to clean the picker
  4. Re-run the Step 04 auth command
Error 05 — PERMISSION_DENIED (API not enabled)
PERMISSION_DENIED: Google Analytics Admin API has not been used in project … or it is disabled
Returned the first time you call get_account_summaries — also appears for the Analytics Data API
Fix
Enable the API in the same project that owns your client_secrets.json

Open both links, confirm the right project is selected at the top, click Enable:

  • Analytics Admin API
  • Analytics Data API

After enabling, no Claude restart is needed — just re-ask the question.

Error 06 — Empty property list / wrong account
analytics-mcp connects but get_account_summaries returns no properties
Empty response OR returns properties from a different Gmail than the one you expected
Fix
You authenticated with the wrong Gmail — your GA properties live under a different account

See the ↺ Switch Account section above. Quickest swap:

powershell
gcloud auth application-default login --client-id-file="C:\Users\YOUR_USERNAME\Documents\client_secrets.json" --scopes=https://www.googleapis.com/auth/analytics.readonly,https://www.googleapis.com/auth/webmasters.readonly,https://www.googleapis.com/auth/cloud-platform

Sign in with the Gmail that owns or has Viewer+ access to the GA4 property you want. Restart Claude Desktop afterwards.

Error 07 — analytics-mcp.exe not found
spawn analytics-mcp.exe ENOENT
Claude Desktop logs show ENOENT when trying to launch the MCP server — OR the hammer icon doesn't appear
Fix
The command path in the config doesn't match where pip actually installed the exe

Find the real path with:

powershell
pip show -f analytics-mcp | Select-String -Pattern "Location:|analytics-mcp.exe"

Combine the Location's parent folder with Scripts\analytics-mcp.exe — that absolute path goes into the command field of the config. If Python is 3.11 the folder is Python311, not Python312.

Error 08 — Credentials file missing
Could not automatically determine credentials
google.auth.exceptions.DefaultCredentialsError — file not found at the GOOGLE_APPLICATION_CREDENTIALS path
Fix
Either the auth command was interrupted, or the path in the config doesn't match where gcloud actually saved the token

Confirm the file exists:

powershell
Test-Path "$env:APPDATA\gcloud\application_default_credentials.json"

If False, re-run Step 04 and wait until PowerShell prints Credentials saved to file: […] before closing anything. If True, make sure the exact same path is in the config's GOOGLE_APPLICATION_CREDENTIALS field (with every backslash doubled in JSON).

Two rules that always apply when using both MCPs together in a single chat.

Multiple accounts — the alias is required on every call.

★ gsc-multi takes the alias, never the email
